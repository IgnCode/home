    Introducción

    Bienvenido al catálogo de microservicios Kubernetes del equipo de UATs y mantenimiento.

    MS Log

    El microservicio Log permite a las aplicaciones implementar fácilmente la funcionalidad de logging a stdout, schema log de base de datos, blob storage... [Enlace a Gitlab].

    Endpoints

    Puedes apagar el microservicio en cualquier momento enviando una petición POST al endpoint /shutdown o comprobar si está desplegado enviando una petición GET a /ready.

    curl -X POST http://127.0.0.1:9009/shutdown

    from requests import post

# Encuesta para no operar hasta que el microservicio esté listo.
api_ready_flag = False
while not api_ready_flag:
    try:
        response = get('http://127.0.0.1:9009/ready')

        if response.status_code == 200:
            api_ready_flag = True

    except exceptions.RequestException as e:
        print('No se puede alcanzar')
        sleep(5)

# Petición para apagar el microservicio       
response = post('http://127.0.0.1:9009/shutdown')

# Comprobación.
if response == 200:
    print("Microservicio Log finalizado\n")

# Envío de la petición de escritura en log.

    Atributo Tipo Descripción

    El microservicio de log está pensado para ser desplegado en de 1 a n contenedores dentro de un pod kubernetes (job, cronjob y derivados...). Para ello, debe ir acompañado de un ConfigMap que le proporcione los parámetros de configuración como variables de entorno y un Secret, para suministrarle los credenciales de acceso a base de datos y/o storage account en su caso.

    ConfigMap

    Parámetro Defecto Descripción
    API_HOST_LOG "0.0.0.0" Dirección IP en la que escucha el microservicio.
    API_PORT_LOG "9009" Puerto en el que escucha el microservicio.
    AZURE_TENANT_ID_LOG "" Tenant id del storage account en el que se almacenarán los mensajes.
    AZURE_STORAGE_ACCOUNT_LOG "" Nombre de la storage account.
    AZURE_STORAGE_CONTAINER_LOG "" Container dentro de la storage account donde se guardarán los mensajes como blob.
    AZURE_LOG_FILE_LOG "" Nombre que se le quiera dar al blob de mensajes.
    namespace "" Namespace de Kubernetes en el que se desplegarán los recursos del microservicio.
    SQL_DB_HOST "" tes
    SQL_DB_DATABASE "" tes
    TO_BLOB_LOG "False" Con valor a True activa en el servicio la opción de guardado de mensajes en blob.
    TO_STDOUT_LOG "False" Con valor a True activa en el servicio la opción de imprimir mensajes.
    TO_FILE_LOG "False" Con valor a True activa en el servicio la opción de guardar mensajes en archivo del directorio local del servicio.
    TERMINATION_PATH_LOG "" Ruta para el archivo de terminación para Kubernetes (debug del servicio).

    Secret

    Parámetro Defecto Descripción
    namespace "" Namespace de Kubernetes en el que se desplegarán los recursos del microservicio.
    AZURE_CLIENT_ID_LOG "" Texto.
    AZURE_CLIENT_SECRET_LOG "" Texto.
    SQL_DB_USERNAME "" test
    SQL_DB_PASSWORD "" tes

    Permisos necesarios

    Dependiendo de la funcionalidad del servicio que queramos utilizar, necesitaremos ciertos permisos proporcionados por Arquitectura/CSC. En caso de que queramos que los mensajes se guarden en el schema log de base de datos, necesitaremos un usuario y contraseña para escribir en esas tablas. En el caso de querer guardar los mensajes en un blob, necesitaremos un service principal que pueda escribir en ese storage account.

    Funcionalidad Permiso Descripción
    Escritura en Blob Storage Account Contributor Permisos de escritura/lectura en storage account.
    Escritura en base de datos READ/WRITE Permisos de escritura/lectura en storage account.

    MS Match

    El microservicio Match nos permite desplegar la funcionalidad de comparación de dos conjuntos de datos.

    Endpoints

    curl -X POST http://127.0.0.1:9009/shutdown

    from requests import post

body = {"nombre": "comparacion 001",
            "ruta_a": "data/a/archivo_a.tsv",
            "ruta_b": "data/b/archivo_b.tsv",
            "pivotes": ["Id", "Match_Id"],
            "ruta_resultado": "Informe_comparativa.xlsx"}

head = {"Content-Type": "application/json"}

response = post(match_api_url+'/comparison', 
                json=body, 
                headers={"Content-Type": "application/json"})

if response.status_code == 200:
  print("[{}][DEBUG] Recibida respuesta afirmativa de ultima peticion de comparacion\n".format(f"{datetime.now():%Y-%m-%d %H:%M}"))

    POST /comparison

    Cuerpo de la petición

    Parámetro Defecto Descripción
    nombre "" (opcional) Nombre de la comparativa.
    ruta_a "ruta/a/archivo_a.tsv" Ruta local al microservicio (ruta al volumen compartido) de donde leerá el conjunto de datos A.
    ruta_b "ruta/a/archivo_b.csv" Ruta local al microservicio (ruta al volumen compartido) de donde leerá el conjunto de datos B.
    pivotes [""] Lista de nombres de columnas por las que se emparejarán los conjuntos de datos.
    ruta_resultado "" Ruta local (ruta al volumen compartido) donde se guardará el informe excel de los resultados.

    Configuración del servicio

    # Contenido de archivo: configmap.yaml
apiVersion: v1

metadata:
  name: match-config
  namespace: ">>Añadir<<"

  TERMINATION_PATH_MATCH: "/usr/src/app/match.log" [opcional]
  API_HOST_MATCH: "0.0.0.0"
  API_PORT_MATCH: ">>Añadir<<"

# Contenido de archivo: app.yaml
apiVersion: v1
kind: Pod
metadata:
  name: app
  namespace: ">>Añadir<<"
spec:
  volumes:
    - name: data-volume
      emptyDir: {}
  containers:
  - name: match-ctr
    image: local/uat-match-ms:0.1
    terminationMessagePath: "/usr/src/app/match.log"
    volumeMounts:
      - mountPath: /usr/src/app/data
        name: data-volume
    envFrom:
    - configMapRef:
        name: match-config

    MS Copy

    El microservicio Copy nos permite realizar peticiones de copiado de archivos de blob storage a local, de local a blob storage, entre blobs storage y de local a local.

    Configuración del servicio

    # Contenido de archivo: configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: copy-config
  namespace: ">>Añadir<<" 
data:
  TERMINATION_PATH_BLOB: "/usr/src/app/copy.log" [opcional]
  AZURE_TENANT_ID_BLOB: ""
  API_HOST_BLOB: "0.0.0.0"
  API_PORT_BLOB: ">>Añadir<<"

# Contenido de archivo: secrets.yaml
apiVersion: v1
kind: Secret
metadata:
  name: copy-secrets
  namespace: etlcal
type: Opaque
stringData:
  AZURE_CLIENT_ID_BLOB: ">>Añadir<<"
  AZURE_CLIENT_SECRET_BLOB: ">>Añadir<<"

# Contenido de archivo: app.yaml
apiVersion: v1
kind: Pod
metadata:
  name: app
  namespace: ">>Añadir<<"
spec:
  volumes:
    - name: data-volume
      emptyDir: {}
  containers:
  - name: copy-ctr
    image: local/uat-match-ms:0.1
    imagePullPolicy: IfNotPresent
    terminationMessagePath: "/usr/src/app/copy.log"
    volumeMounts:
      - mountPath: /usr/src/app/data
        name: data-volume
    envFrom:
    - configMapRef:
        name: copy-config

    Parámetros de ConfigMap copy-config:

    Parámetro Defecto Descripción
    namespace "" Namespace asignado por Arquitectura dentro del cluster Kubernetes.
    API_HOST_BLOB "0.0.0.0" Dirección IP en la que el microservicio escucha las peticiones.
    API_PORT_BLOB "9009" Puerto TCP en el que el microservicio escucha las peticiones.
    TERMINATION_PATH_BLOB "/ruta/a/archivo.log" (Opcional) Ruta para escribir el archivo de depuración interno del microservicio.
    AZURE_TENANT_ID_BLOB "" El tenant id del service principal proporcionado por Arquitectura.

    Parámetros de Secret copy-secrets:

    Parámetro Defecto Descripción
    AZURE_CLIENT_ID_BLOB "" Client ID del service principal con acceso a los storage account, proporcionado por Arquitectura.
    AZURE_CLIENT_SECRET_BLOB "" Secreto inyectado por Jenkins en el momento del despliegue. Arquitectura propociona un alias.
    You must replace meowmeowmeow with your personal API key.

    POST /copy

    Body de la petición:

    Parámetro Ejemplo Descripción
    from_blob "/storage-account/container/ruta/a/blob.tsv" Ruta al blob origen, empezando por el nombre del storage account.
    from_local "ruta/a/archivo.txt" Ruta al archivo oriden, del sistema de ficheros local al microservicio (por ejemplo, un archivo de un volumen compartido).
    to_blob "/storage-account/container/ruta/a/blob.dat" Ruta al blob que queremos crear, empezando por el nombre del storage account.
    to_local "ruta/a/archivo.txt" Ruta local al microservicio donde queremos escribir el archivo.

    Para refrescar un informe del servicio online de PowerBI, necesitamos conocer el nombre del informe desplegado y el nombre del workspace en el que se ha desplegado.

    Solicitud HTTP

    POST http://127.0.0.1:PUERTO/refresh

    Cuerpo de la solicitud HTTP

    Parámetro Tipo Descripción
    workspace String Nombre del workspace.
    informe String Nombre del informe.
    Recuerda — en la petición debes utilizar el puerto al que has configurado el servicio para que escuche.

    Service principal

    Para ejecutar la pieza, necesitamos suministrarle los credenciales de un service principal con los siguientes permisos.

    Permiso Descripción
    Power BI Service.Group.Read Permisos de lectura en los grupos a los que pertenece.
    Power BI Service.Group.Read.All Permisos de escritura/lectura en storage account.
    Power BI Service.Workspace.Read.All Permisos de escritura/lectura en storage account.
    Power BI Service.Report.Read.All Permisos de escritura/lectura en storage account.
    Power BI Service.Dataset.Read.All Permisos de escritura/lectura en storage account.

    Errors

    This error section is stored in a separate file in includes/_errors.md. Slate allows you to optionally separate out your docs into many files...just save them to the includes folder and add them to the top of your index.md's frontmatter. Files are included in the order listed.

    The Kittn API uses the following error codes:

    Error Code Meaning
    400 Bad Request -- Your request is invalid.
